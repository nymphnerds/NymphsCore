// What makes a useful report

Small details make bugs fixable.

Polished writing is not required. Specifics are. The fastest fixes usually come from the tester who recorded the exact button, step, message, or moment where trust dropped.

  • State the exact step or panel, not just the feature area.
  • Tell us what you expected before the workflow went wrong.
  • Paste logs when install, model fetch, or runtime start fails.
  • Say if the issue felt broken, confusing, scary, or simply too slow.
Good report title Manager smoke test says ready, but TRELLIS start still fails on first click.
Bad report title It kind of broke and I do not know why.
// Send alpha feedback

Report the exact break point.

Submissions go straight to the NymphsCore alpha feedback inbox. Discord is the fastest place for follow-up and tester coordination.

If the issue happened during install or runtime start, include the moment it first became confusing, not just the final error line.

// What to actively push on

Do not just test success.
Test confusion and trust.

The strongest alpha notes often come from moments where the product technically works, but the tester still feels lost. That is a product problem too.

First-run clarity

Could you tell what to do next at every step without asking for hidden context?

Repair paths

Try interrupted installs, missing models, backend checks, and reruns. The recovery path matters.

Speed expectations

Report where progress feels fake, stalled, or too silent to trust.

Mental model

Call out places where Manager, Blender, runtime tools, and generated assets stop fitting together in your head.

// Fast rubric

What we care about most

If you only have time for one solid note, prioritize the place where the workflow most damaged confidence.

  • What was the first confusing or misleading signal?
  • What assumption did the product make that a tester would not naturally make?
  • What single UI, copy, or default change would have saved the session?
  • What unexpectedly worked well and should survive later cleanup?
// Final note

Break it honestly.
Help us fix it faster.

Alpha feedback is most valuable when it names the exact point where the workflow stopped feeling trustworthy. That can be a crash, a missing model, a confusing panel, a silent delay, or a step that only makes sense if a developer explains it out loud.
