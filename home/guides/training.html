Guide 05

Z-Image TurboAI Toolkit LoRA Training Guide

Prepare the dataset Choose settings Advanced AI Toolkit

This guide follows the real training flow in NymphsCore: prepare the dataset, optionally draft captions with Brain, choose the training settings, click Add Job, then click Start Job. The Manager is just the front end. AI Toolkit is the real job runner underneath.

  • Good images and a tight visual target matter more than any single magic setting.
  • Caption with Brain is optional and comes after the dataset exists.
  • Add Job happens after you decide the settings, not before.
  • Start Job only runs the saved AI Toolkit job.
  • The first run is about getting signal, not perfection.

The shortest honest version is this: put the right images in first, keep the dataset coherent, use captions as support instead of as the main event, and judge the run by its samples and checkpoints rather than by theory.

  • Use clean images that all teach the same idea.
  • Use captions to describe what is in each image when you need them.
  • Use Fast Test for quick signal and Baseline for the safest normal first run.
  • Use the advanced section at the end only if you want to stay inside the upstream AI Toolkit UI.

1. Prepare the dataset first.

Start by creating or picking the dataset name, then use Open Pictures Folder and put the training images there. The guide makes more sense once the dataset already exists, because the rest of the flow depends on those images being in place.

What to use: aim for clean images that hold up at the trainer's current 1024 resolution target. Larger originals are fine. Roughly speaking, images that are at least around 1024 pixels on the short side are the easiest starting point.
What to avoid: blurry grabs, heavily compressed thumbnails, mixed-quality scraps, and random near-duplicates usually add more noise than value. A smaller clean set is usually better than a larger messy one.
For style LoRAs: keep the dataset visually coherent. If the images do not feel like they belong to one family, the LoRA has a much harder time settling into a reliable look.

2. Optionally draft captions with Brain, then review them.

After the dataset exists, you can use Caption with Brain if you want help drafting captions. This is optional. It is not the first step of the workflow, and it is not a substitute for good images or sensible settings.

What Brain does: it drafts one editable caption per image into metadata.csv. The UI lets you choose Fill Blank Only or Overwrite All Drafts, then review the result through Open Draft CSV.
What Add Job does later: when you click Add Job, the app mirrors those draft captions into the per-image .txt files that AI Toolkit actually trains from. That means metadata.csv is the sheet you edit, but the final training text lives beside the images.
How to caption style work: describe the content normally instead of padding every line with repeated style labels. The images should teach the rendering style. The captions should teach what to ask for. If you have 10 images, you should end up with 10 matching caption rows.

3. Decide the training settings before you click Add Job.

This is the part that comes before job creation. In the Manager, you choose the preset, adapter, sample prompt, steps, checkpoints, learning rate, rank, and whether to use Low VRAM mode. If you change those later, the app will tell you to click Add Job again before it lets you start.

Preset choice: Fast Test is the quickest path when you just want to see whether the LoRA is learning at all. Baseline is the safest normal first run. Style is the normal style preset. Strong Style is the follow-up when the normal style run still feels too polite.
Adapter choice: stay on v1 (Recommended) for the first serious run. Use v2 (Experimental) only after the normal path is behaving and you actually want a comparison.
Sample prompt: this matters more than it looks. AI Toolkit uses it to generate preview samples while the job is running, so it should reflect the kind of image you actually want to judge during training.
Simple rule: choose the smallest set of settings that answers the question you have. If you only want to know whether the dataset works at all, use the quick path. If you want the safest normal proof, use Baseline. Save heavier style pushes for later.
Job name and trigger word: in the AI Toolkit transcript, the training name is just the LoRA/job name. A trigger word is optional. If you use one, keeping the job name and trigger word close to each other makes it easier not to forget what to prompt later.
Steps, save every, and checkpoints: steps are the total length of the run. Save every controls how often a checkpoint is written. Max steps to keep saved controls how many of those checkpoints survive instead of being overwritten. In plain English: longer training and more saved checkpoints give you more chances to find the sweet spot, but they also cost more time and disk space.
Batch size and gradient accumulation: batch size is how many images the job sees at once. Bigger batches usually give the model a better sense of the concept, but they use more VRAM. Gradient accumulation is the slower workaround when you want batch-like behavior without actually fitting that bigger batch in memory.
Optimizer, learning rate, and weight decay: the transcript treats AdamW8Bit as the normal default when VRAM allows it. Learning rate is how hard each step pushes the LoRA. Too low and it barely learns. Too high and it overlearns or breaks down. Weight decay is a light anti-overcooking control that helps stop the model from clinging too hard to exact training images.
Timestep type and bias: these control where training pays attention across the denoising process. Balanced is the normal safe start. High noise is more about earlier composition-building stages and can help when you are trying to force a much stronger overall style shift. That is why the Z-Image Turbo transcript starts balanced and only later leans harder into high noise.
Text encoder and unload text encoder: the text encoder is what helps the model understand prompt words more specifically. Turning it off or unloading it is mostly a VRAM tradeoff. If you are memory-constrained, unloading it can help, but it slows training down because it keeps having to reload that component.

4. Click Add Job, then Start Job.

These are two separate steps. Add Job is the commit point. Start Job is the run point. The app is designed around that distinction.

Add Job: prepares the dataset support files, refreshes metadata.csv, mirrors the captions into per-image .txt files, writes the AI Toolkit YAML job, writes the LoRA metadata, and registers the job with the official AI Toolkit UI.
Start Job: queues and starts the saved AI Toolkit job. If you changed settings after the last Add Job, the app will stop you and tell you to apply those pending changes first.
Real Flow
1. Pick the dataset / LoRA name 2. Open Pictures Folder and add the images 3. Optionally use Caption with Brain 4. Review metadata.csv if needed 5. Choose preset, adapter, sample prompt, and training settings 6. Click Add Job 7. Click Start Job

5. Use samples and checkpoints to judge the run.

The first run does not need to be perfect to be useful. What matters is whether the samples and checkpoints show the LoRA moving in the right direction. That is the signal you use to decide whether to keep, adjust, or rerun.

What success looks like: the job finishes cleanly, writes checkpoints or a final LoRA, and the training samples visibly move toward the intended character, style, or concept. Halfway there is still useful information.
What weak results usually mean: the dataset may need to be cleaner, the style signal may need a stronger preset or longer run, or the sample prompt may not be showing the thing you actually care about.
What overcooked results usually mean: do not just crank it harder. Look for noisy captions, mixed images, or a dataset that is trying to teach too many different things at once.
Checkpoint testing: start with the later checkpoint or final file first, then compare backward if it feels too weak or too aggressive.

6. Test the LoRA in Blender.

Once the run is done, the Blender side should be simple: enable the LoRA, pick the checkpoint or final file, prompt normally, and only start comparing variations after you have seen one clean first result.

First test: choose the LoRA in the Z-Image panel, leave the recommended activation on for the first run, and generate once before changing several things at the same time.
Prompting: prompt the scene or subject normally. The activation phrase is only there to help some LoRAs lean into their learned look more clearly if they need that extra nudge.
Comparison order: if the result is weak, keep activation on first, then adjust LoRA strength, then compare checkpoints. If the result is already strong, you can try turning activation off to see whether it is still needed.

7. Advanced: if you want to do the whole run in the AI Toolkit UI.

The local transcripts show a very consistent upstream AI Toolkit flow: create the dataset, add images, optionally add captions, create the job, choose the architecture and adapter, set the sample prompts and job settings, create the job, then go to the training queue and start it. This section is for advanced users who want the setting meanings directly instead of the simplified guided flow.

Transcript-backed starting point: for the AI Toolkit UI, a common starting direction in the transcripts is AdamW8Bit, 1e-4 learning rate, weight_decay 1e-4, weighted timesteps, MSE loss, EMA off, cache text embeddings on, and a balanced bias before moving into a stronger style push later.
What those settings mean: learning rate is how hard each step pushes the LoRA; weighted timesteps control where the job pays attention across the noise schedule; balanced means you are not immediately over-biasing toward the highest-noise part of the process; MSE is the normal loss choice here; EMA off and cache text embeddings on are part of the common baseline rather than special expert tricks.
What the Z-Image Turbo transcript adds: for extreme style changes, Ostris starts from a balanced setup and then later switches toward a high noise bias when he wants the model to rebuild more of the early composition. That is useful later, but it is not the first thing to reach for on every run.
Batch size and gradient accumulation: in the AI Toolkit settings transcript, batch size is how many images the job sees at once. Gradient accumulation is the slower workaround when you want batch-like behavior without the VRAM for a larger real batch.
Trigger words and captions: the transcripts do not treat trigger words as universal law. Some runs use them, some do not. For style work, the transcript examples lean toward captioning the content normally and letting the images teach the style.
Direct AI Toolkit UI Order
1. Datasets -> New Dataset 2. Add the images 3. Add captions if you want them 4. New Job 5. Set training name and trigger word if using one 6. Select Z-Image Turbo and the training adapter 7. Set prompts and training settings 8. Create Job 9. Training Queue -> Start 10. Judge samples and checkpoints
