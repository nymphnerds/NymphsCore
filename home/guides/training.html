Guide 05

Z-Image TurboAI Toolkit LoRA Training Guide

Understand captions Run your first test Back to guide hub

The trainer now runs through a managed AI Toolkit sidecar, and the Manager treats captions as user-owned training data. Nymphs Brain can help draft them, but you still review what will actually train.

  • The Manager prepares the dataset and job, then hands training off to AI Toolkit.
  • Fast Test is the shortest path when you just want a quick training signal.
  • Baseline is the safe normal first preset for proving the end-to-end flow.
  • Style and Strong Style are the current style presets in the Manager.
  • v1 is the recommended adapter for first tests. Save v2 for comparison after the stable path works.
  • Manager now saves a recommended activation with the LoRA, and Blender can use it automatically.

The images teach the model what things look like. The captions teach it what words should call those things back up later. Then, after training, some LoRAs also respond better when Blender includes a short activation phrase like the LoRA name itself.

  • The caption is not busywork. It is part of the training pair.
  • You do not need perfect prose. You do need clear intent.
  • Caption Brain is a draft helper, not a final authority.
  • For style work, describe the subject normally and let the LoRA learn the rendering style.
// Captions

What captions are actually doing.

A caption is the text side of the training example. When the trainer sees an image of a bear in a childlike crayon drawing and the prompt says a bear building a log cabin in the snow, it learns how that subject and wording connect. Later, when you prompt for a bear building a cabin, the LoRA has something meaningful to grab onto.

Most important beginner rule: you do not write one caption for the whole folder. You write one prompt for each image. So if you put in 10 images, you end up with 10 rows in metadata.csv.
// Direct Answer

Do I need one prompt per image?

Yes. The simple beginner rule is: if you have ten training images, write ten prompts. Each image should have one matching line in metadata.csv.

// Beginner Rule

Keep each caption simple.

You are not writing a novel or technical report. One clear line per image is enough. Think of it as the prompt you would have wanted if you were trying to recreate that picture later.

Wrong idea

You have 10 images in the folder and write 1 prompt for all of them together.

Do not do this. The trainer needs to know which prompt belongs to which exact picture.

Right idea

You have 10 images in the folder and write 10 prompts total.

Each row in metadata.csv matches one image to one prompt.

Rule 01

Describe the subject, action, and key read.

Write what is in the image in the way you would want to prompt it later. This is usually more useful than stuffing in every technical detail you can think of.

  • Good: a child holding a red kite in a windy field
  • Good: a rabbit reading a book at a wooden desk
  • Weaker: nice art style cute image
Rule 02

Use Caption Brain as a draft assistant.

Nymphs Brain can draft one prompt per image into metadata.csv. That saves time, but the training text is still yours to review and edit before you start a run.

  • The Manager keeps the image rows in sync for you.
  • Caption Brain can fill blanks or rewrite draft rows.
  • You should still fix anything that reads vague, wrong, or noisy.
Rule 03

For stylized training, caption the content normally.

This is the big idea from the kids-drawing style workflow. If every caption just repeats the style name, the LoRA can become less useful. For strong styles, it is often better to caption the content cleanly and let the LoRA absorb the visual treatment from the images themselves.

Example

Kids drawings caption example.

Imagine the image is a crayon-like drawing of a bear building a cabin in snow. A useful caption is usually about the scene, not a constant reminder that it is a kids drawing.

  • Better: a bear building a small log cabin in the snow
  • Also good: a cartoon bear building a wooden cabin during snowfall
  • Weaker: kids drawing style crayon style cute style bear
Finished Example · metadata.csv

That is the shape you are aiming for: one image name, one prompt, repeated all the way down. It does not need to sound fancy. It needs to match the picture.
Plain-English Translation
Image 1 = bear_cabin_01.png Prompt 1 = "a bear building a small log cabin in the snow" Image 2 = rabbit_book_01.png Prompt 2 = "a rabbit reading a book at a wooden desk" Image 3 = girl_kite_01.png Prompt 3 = "a child holding a red kite in a windy field" Image 4 = cat_bike_01.png Prompt 4 = "a cat riding a bicycle down a quiet street" Keep going like that until every image has its own matching prompt.
If you are unsure what to write, aim for one clear sentence per image. Subject first, then action, then the most important visible qualifiers. For kids-drawings style, think: prompt the scene normally, and let the images teach the childlike rendering.
// Source Images

Use clean images that fit the current trainer.

The current Z-Image Trainer writes AI Toolkit jobs at 1024 resolution. That does not mean every source file must already be exactly 1024 by 1024, but it does mean the dataset works best when the images are clean, reasonably detailed, and not tiny web thumbnails full of compression damage.

// Best Practical Size

Aim for images that hold up at 1024.

As a practical rule, use source images that are at least around 1024 pixels on the short side when you can. Larger originals are fine. Very small images can still be used, but they usually do not teach clean detail because the trainer cannot invent information that is not there.

// Aspect Ratio

Clean crops beat awkward padding.

The trainer handles resizing for you, but wildly mixed aspect ratios can make the dataset feel less coherent. Try to keep the subject framed clearly, crop away dead space when it is not helping, and avoid relying on ugly borders or screenshots with UI chrome still visible.

Rule 04

Do not train on junk just because you have it.

Low-resolution thumbnails, heavily compressed exports, blurry grabs, and random near-duplicates usually add more noise than value. A smaller clean set is usually better than a larger messy one.

  • Good: clear originals, thoughtful crops, consistent visual quality
  • Risky: tiny marketplace previews, screenshots, or mixed-resolution scraps
  • Bad: images where the subject is hard to read even to you
Rule 05

Keep content and style consistent enough to teach one idea.

If you are training a style LoRA, the dataset should still feel like it belongs to one visual family. Mixed mediums, mixed rendering quality, and mixed subject treatment make it harder for the LoRA to settle into a reliable look.

Current system detail: the Manager keeps metadata.csv as the editable caption source, then mirrors those captions to per-image .txt files for AI Toolkit before training. You do not need to hand-build those text files yourself.
// Presets

Pick the preset that matches the job.

The current Manager exposes four preset choices: Fast Test, Baseline, Style, and Strong Style. You do not need to think about every underlying AI Toolkit knob first. Start with the shortest or safest path that answers the question you actually have.

Fast Test: use this when you want a quick signal and do not want to wait through a longer run first. Baseline: use this for the safest normal first proof that the trainer, runtime, and addon are all lined up.
Style: use this for painterly or illustrative datasets. Strong Style: use it as the stronger follow-up when the normal style run still feels too polite.
// Recommended Activation

What recommended activation means.

This is the simple version: some LoRAs work strongly just by being turned on, and some work better when Blender also adds a short phrase to the prompt. NymphsCore calls that phrase the recommended activation so you do not need to think in trigger-word theory first.

// Plain English

Think of it as the extra nudge.

The LoRA is the attachment. The activation phrase is the extra nudge that tells the model to lean into that learned look more clearly.

// Product Rule

You should not have to guess.

Manager now writes a recommended activation into LoRA metadata. Blender reads it and can use it automatically, so the system does the remembering for you.

Rule 11

Recommended activation is a suggestion, not a law.

Some LoRAs look great without it. Some feel weak until you use it. That is why the addon lets you keep it on by default, then turn it off later if the style is already strong enough.

Rule 12

The current default is just the LoRA name.

If your LoRA is called yamamoto, the system now recommends yamamoto as the starting activation text. Keep it simple first, then change it only if testing shows the LoRA wants something else.

Rule 13

Downloaded LoRAs may need a quick manual setup.

If a downloaded LoRA does not already have metadata, you can type the recommended activation once in Blender. The addon saves it beside the LoRA so you do not have to remember it next time.

Current system detail: Manager-trained LoRAs now get a nymphs_lora.json sidecar with activation metadata. Blender reads that sidecar and uses it to fill the LoRA controls automatically.
// Blender

How to use the trained LoRA in Blender.

Once training is done, the normal flow is simple: pick the LoRA in the addon, leave the recommended activation on for the first test, and compare a run with it on versus off only if you need to judge how much it helps.

Step 01

Open the Z-Image panel and enable the LoRA.

Turn on Use LoRA, choose the LoRA run, and select a checkpoint or final file. The addon now lists LoRAs from the LoRA library as well as trainer run folders.

Step 02

Leave recommended activation on for the first test.

The addon shows Use Recommended Activation and Recommended Activation. Start with that enabled so you are testing the LoRA the way NymphsCore currently recommends it.

Step 03

Write the rest of the prompt normally.

You still prompt the scene, subject, or idea you want. The activation phrase is only there to help the model call up the LoRA more reliably when needed.

Step 04

If the result is weak, keep activation on before changing everything else.

First try the recommended activation, then adjust LoRA strength, then compare checkpoints. Do not make five different changes at once or it becomes hard to tell what actually helped.

Step 05

If the result is already strong, try turning activation off.

That is the clean way to judge whether the LoRA really needs the extra phrase or whether it is already asserting the style strongly enough on its own.

Simple Blender Flow
1. Train LoRA in Manager 2. Open Blender 3. Z-Image panel -> Use LoRA 4. Pick LoRA 5. Leave "Use Recommended Activation" on 6. Generate once 7. If needed, compare with activation off
// First Test

Your first run, without assuming flagship hardware.

The goal of your first training run is not perfection. It is to confirm the pipeline works, that your prompts make sense, and that the LoRA is moving in the right direction before you invest in a longer run. Smaller GPUs may need more patience, smaller image sets, and more conservative expectations, but they can still learn useful things.

// Hardware Reality

You do not need the biggest card to start.

If your GPU is more modest, treat a small Baseline or Style run as your friend. Use a smaller, tighter dataset first and judge whether the LoRA is learning at all before you chase strength.

// What To Expect

Longer waits do not automatically mean failure.

The first install or first model load can take a while. The trainer now pulls the heavy base trainer files up front, but the first real run can still be quiet for a bit while the model settles into memory.

Step 01

Start with a tiny, consistent dataset.

Use roughly 8 to 20 images for the first pass. Keep the style tight, prefer clean crops, and avoid mixing wildly different aesthetics together.

Step 02

Pick the right focus before you pick strength.

Choose Baseline for the safest first proof. Choose Style for painterly or illustration-style work. Save Strong Style for the second pass if the style still feels too polite.

Step 03

Review every prompt before you hit start.

Use Open Pictures Folder, then Open Draft CSV, and make sure every image has a real prompt. If you use Caption Brain, scan for anything too generic, repetitive, or simply wrong.

Step 04

Treat the first run as a learning run.

If it completes and the result bends output toward your style, that is a win. Stay on v1 (Recommended) for the adapter first. Move to Strong Style or try v2 (Experimental) only after the baseline path is behaving.

Step 05

Use Low VRAM mode when the hardware needs it.

Low VRAM mode exists to keep borderline cards in the game. It is a practical stability switch, not a badge of failure. If your GPU is tight on memory, use it early instead of wasting runs on avoidable crashes.

// Results

How to read what comes out.

The first LoRA does not need to be perfect to be useful. What matters is whether it is pointing in the right direction and whether the failure mode tells us what to change next.

// Healthy Signals

What success looks like.

A healthy first run finishes cleanly, writes a LoRA file, and clearly nudges output toward the intended look. Even if it only feels halfway there, that is enough information to improve the next run.

// Trouble Signals

What failure usually means.

If the result is weak, the style signal may need more data or more training. If it becomes mushy, strange, or too aggressive, the dataset may be messy or the run may be pushing too hard for the data quality.

First Check

Compare the later epoch first.

The trainer writes multiple checkpoints while it trains, plus the final LoRA file in the LoRA folder. Start by testing the later checkpoint or final file first, then compare backward if it feels too weak or too pushy.

If It Is Weak

Try a cleaner or slightly longer second pass.

Move from Baseline to Style, or from Style to Strong Style, tighten prompts, and make sure the images are all really teaching the same visual language. Also test the recommended activation in Blender before assuming the LoRA itself failed.

If It Is Overcooked

Do not just crank it harder.

Look for noisy prompts, mixed styles, or images that do not belong in the set. A stronger run on a confused dataset usually gives a stronger mess.

// Advanced

If you want to do the whole run in the AI Toolkit UI.

This page is still written around the simpler guided flow, but some of the transcript research is more useful if you are working directly in the upstream AI Toolkit UI. Treat this as a practical starting point for advanced users, not as a law that every dataset must follow exactly.

Good starting direction from the transcript: AdamW8Bit, 1e-4 learning rate, weight_decay 1e-4, weighted timesteps, MSE loss, EMA off, cache text embeddings on, and a more balanced style/content setup before you move into a stronger style push.
What those settings mean: learning rate is how hard each step pushes the LoRA; weighted timesteps decide where training attention lands across the noise schedule; balanced style/content is the safer middle ground before going more aggressive; MSE is the normal loss choice here; EMA off and cache text embeddings on are part of the current recommended baseline rather than special expert tricks.
The practical workflow idea: start with a sane baseline that gives you signal, then push harder only if the result is still too weak or too polite. For style work, keep the captions focused on the image content instead of stuffing them with repeated style labels. The images should teach the rendering style. The captions should teach what to ask for.
Two easy mistakes to avoid: do not assume a trigger phrase is mandatory for every LoRA, and do not assume stronger settings will rescue a messy dataset. The transcript notes and local testing both point the same way: clean images, honest captions, and a balanced first pass matter more than trying to win with one magic knob.
