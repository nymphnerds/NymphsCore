# Retexturing User Guide

This note tracks the current retexturing workflows in the Blender addon and how to choose between `2mv` and `2.1`.

## Quick Guidance

- use `2mv` for the practical everyday retexture workflow
- use `2.1` texture-only when you specifically want the heavier PBR-oriented path
- keep normal `2.1` for shape generation
- treat normal `2.1` shape+texture as the heavier path, not the practical default on a `16 GB` GPU

The current reality is simple: `2mv` is the stronger practical retexture lane today. Newer is not automatically better here. `2.1` matters because it exposes the public PBR-oriented path, but `2mv` is still the workflow that feels better for repeated real use.

## Current Selected-Mesh Workflow

The intended retexture workflow is:

1. Select a mesh in Blender.
2. Provide a guidance image.
3. Run `Retexture Selected Mesh`.

That path is available in both `2mv` and `2.1`, but the two backends have different strengths.

## `2mv` / `2.0` Retexture

This is still the more practical day-to-day lane.

- best general-purpose selected-mesh retexture path
- faster and lighter than `2.1`
- good fit for mixed-source meshes, including meshes generated by `2.1`
- preferred when speed and reliability matter more than PBR richness

### Current `2mv` Texture Controls

- `Face Limit`
- `Texture Size`

Current `Texture Size` values:

- `512 px`
- `1024 px`
- `2048 px`

### `2mv` Startup / Model Controls That Affect Texture

- `Texture`
- `Turbo`
- `FlashVDM`

Practical reading:

- `Turbo` is the bigger speed vs quality tradeoff
- `FlashVDM` is mainly a shape-side acceleration feature and matters less for selected-mesh retexture than `Turbo`

## `2.1` Retexture

`2.1` is the higher-end path, mainly because of the PBR texture workflow.

- better suited to high-quality PBR-oriented retexture work
- much heavier than `2mv`
- can be too much for a normal `2.1` shape+texture server on a `16 GB` GPU
- now supports a separate `2.1` texture-only server path for selected-mesh retexture

The dedicated `2.1` texture-only server was the breakthrough that made full `2.1` PBR retexture complete on a `16 GB` GPU during local testing.

### Current `2.1` Texture Controls

- `Face Limit`
- `Max Views`
- `Texture Size`
- `Remesh Uploaded Mesh`

Practical reading:

- `Max Views` is one of the main quality/runtime knobs
- `Texture Size` trades VRAM/runtime against output quality
- `Remesh Uploaded Mesh` is experimental and should be treated as a rescue option, not the default
- for many meshes, preserving uploaded topology works better than remeshing first

## Which One To Use

### Use `2mv` when:

- you want the practical default retexture workflow
- you want faster runtime and lower VRAM pressure
- you are working with mixed-source meshes and want the safer lane

### Use `2.1` texture-only when:

- you specifically want the PBR-oriented path
- you are willing to accept heavier runtime
- you want the dedicated retexture-only server instead of carrying the shape model in memory

## Future Direction

The current texturing split does not have to be the final one.

- `2mv` can remain the practical default even if a future backend becomes the better high-end texture path
- `2.1` texture-only is the current public PBR lane, but it may eventually be replaced or complemented by a completely separate texture backend
- if a stronger public texturing backend appears later, Nymphs3D can keep `2mv` / `2.1` shape generation while swapping in a different texturing engine

## Input Notes

For `2.1` shape generation, transparent PNGs are not always equal. A source can be genuinely RGBA and still have very soft alpha around the subject footprint. In those cases:

- `Auto Remove Background` may still produce cleaner shape results than trusting the stored alpha
- disabling background removal can sometimes introduce a support/base plane if the alpha edge is too soft

For selected-mesh retexture, this matters less than it does for image-to-shape, but clean source images still help.
